Segurança e Sandboxing: Concedendo Poder com CautelaO poder de um agente de IA que pode ler e executar código no sistema local cria uma superfície de ataque expandida, tornando a segurança uma preocupação primordial.18 O Codex CLI aborda essa questão com um modelo de segurança robusto baseado em sandboxing.9 O mecanismo de endurecimento se adapta ao sistema operacional do host: no macOS, ele utiliza a tecnologia nativa Apple Seatbelt (sandbox-exec); no Linux, a recomendação é utilizar Docker para isolamento. O suporte para Windows é experimental e é realizado através do Windows Subsystem for Linux (WSL2), que por sua vez se beneficia da abordagem de sandboxing do Linux.9O sandboxing mitiga riscos de execução de código malicioso, mas não elimina todos os vetores de ataque. A injeção de prompt (prompt injection) é uma vulnerabilidade inerente aos modelos de linguagem, onde entradas maliciosas disfarçadas de prompts legítimos podem manipular o modelo para executar ações indesejadas, como a exfiltração de dados sensíveis ou a transmissão de malware.18 O modelo de segurança do Codex CLI, que depende dos modos de aprovação e do isolamento, não anula a necessidade de vigilância. Um agente de IA com acesso a arquivos locais e capacidade de executar comandos representa um vetor de ataque que exige uma nova forma de monitoramento, pois instruções maliciosas podem ser disfarçadas em prompts aparentemente inofensivos, ignorando as salvaguardas internas do modelo.4.3. # 4) Checklist de segurança operacional e testes de sandbox (Windows & Linux & macOS)
**Referências:** docs + issues + blogposts sobre sandboxing e WSL2. citeturn0search6turn0search20turn0search7

**Checklist mínimo (executável):**
1. **Ambiente de teste isolado**
   - Crie VM/container separado (Docker, WSL2 VM, ou VM dedicada). Não executar Full Access em produção. citeturn0search6
2. **Verificar sandboxing**
   - Linux: executar `codex` dentro de container Docker; testar se o processo tenta acessar rutas fora do container.  
   - macOS: revisar uso do Seatbelt / sandbox-exec conforme docs.  
   - Windows: preferir WSL2 e isolar filesystem; testar issues conhecidas (ver issue #2549). citeturn0search20turn0search6
3. **Validar modos de aprovação**
   - Testar `--suggest`, `--auto-edit`, `--full-auto` em repositório de teste; verificar logs e diffs antes de aplicar. citeturn0search13
4. **Gating via CI**
   - Nunca permitir merge automático de commits gerados por agente sem revisão humana; usar PR templates, CI checks obrigatórios e branch protections. citeturn0search7
5. **Proteção contra prompt-injection**
   - Filtrar/escapar entradas externas (logs, uploads) antes de passá-las ao agente; revisar o conteúdo do prompt para marcações especiais.  
6. **Auditoria e logs**
   - Ativar logs JSON (se disponível) para cada `codex exec` (guardar trajeto, ações propostas e execuções). StackOverflow mostra discussão sobre salvar exec output. citeturn0search4
7. **Segurança de chaves**
   - Não armazenar API keys em plain text; usar secrets manager; revisar permissões do usuário que executa `codex`.  
8. **Teste de reversão**
   - Criar scripts `undo` automáticos (revert git, apply reverse patch) e validar em ambiente de staging.  
9. **Revisão de dependências**
   - Bloquear execuções que baixam pacotes externos automaticamente; exigir revisão humana antes de `npm install` / `pip install` executivos.  
10. **Plano de resposta a incidente**
    - Se agente executar comando malicioso: isolar host, recolher logs, reverter commits e rotacionar chaves expostas.
